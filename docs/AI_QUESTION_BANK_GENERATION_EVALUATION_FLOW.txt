PART A — HOW TO REACH difficultyMatchScore 80+ (PRACTICAL GUIDE)
================================================================================

Your target metric is OpenAI validation difficultyMatchScore (not internal
finalize difficultyMatchScore). Internal finalize often scores 82–94 while
OpenAI validation stays ~64–66 when:

  1. Explanations contain meta draft text ("adjusting", "is incorrect", "Let's use")
  2. OpenAI auditor sees formula plug-ins despite multi-step _solveSteps
  3. A few questions per bank are flagged too-easy (Q1, Q4, Q6 pattern)
  4. Client validate payload may omit _solveSteps (only explanation text sent)

WHAT THE PIPELINE ALREADY DOES (AI STEERING — 3 LLM CALLS PER CHUNK)
--------------------------------------------------------------------
  Call 1 — Archetype planning (Gemini/OpenAI):
    resolveConceptArchetypeSteering()
      → buildArchetypePlanningPrompt()
      → callQuestionBankGenerationLLM()
      → parseArchetypePlanResponse()
    Plans N unique hard slots + per-slot blueprints for the batch.

  Call 2 — Skeleton generation (Gemini/OpenAI):
    buildSolveFirstSkeletonPrompt()
      → callQuestionBankGenerationLLM()
      → parseSolveFirstSkeletons()

  Call 3 — Validation audits (OpenAI default):
    buildTopicRelevancePrompt() + buildCorrectnessAuditPrompt()
      + buildJeeAuthenticityAuditPrompt()
    mergeValidationResults() → difficultyMatchScore

TO ASK AN AI TO FIX LOW DIFFICULTY — USE evaluation_regen (BUILT-IN)
--------------------------------------------------------------------
After validate returns difficultyMatchScore < 80:

  1. Frontend calls validate → gets regenerationInstructions + dimensionScores
  2. User triggers regenerate with:
       generateIntent: "evaluation_regen"
       topicRelevanceEvaluated: true
       topicRelevanceRegenerated: false
       topicRelevanceFeedback: { full validate response payload }
  3. Backend injects into generation prompt:
       buildRegenerationEscalationBlock()
       buildRegenerationQualityGatesBlock()
       formatTopicRelevanceFeedbackBlock()
  4. evaluation_regen uses **solve-first** for STEM singles (same as initial).
     No one-shot fallback on regen — partial batches merge client-side.
  5. Post-skeleton **LLM difficulty self-audit** rejects weak skeletons before MCQ build.
  6. **Targeted regen** replaces only failed question numbers (not full bank upscale).

ALTERNATIVE: ADD A 4TH AI CALL (IMPLEMENTED)
---------------------------------------------
  Post-skeleton, pre-finalize:
    applySkeletonDifficultySelfAuditGate() → LLM scores each skeleton 0–100
    applyDifficultySelfAuditGate() on finalize for built MCQs
  Env: AI_QB_DIFFICULTY_SELF_AUDIT=1 (default on), AI_QB_DIFFICULTY_SELF_AUDIT_MIN
       (default: veteran 80 / non-veteran 75; skeleton bar veteran 78 / non-veteran 70)
  NOTE (2026-07-14): veteran finalize gate lowered 85 → 80 to match the audit
       rubric's own "80+ = clearly meets tier" line. The old 85 bar rejected
       questions the rubric rated fully tier-compliant, dropping good hard items
       on single-shot LLM scoring noise and forcing needless regens. Log analysis
       (runs 07-06→07-10) showed ~52 questions scoring 80–84 dropped solely due to
       the 85 bar, plus frequent 4–8-per-batch rejection waves; validation-side
       difficultyMatch was already mostly 88–100, i.e. not the bottleneck.

THREE GENERATION MODES (generationMode field)
------------------------------------------------
  "default" (aka solve-first / topic-first) — code plans concept slots, LLM
    solves each into a skeleton, code builds the MCQ. See PART F.

  "prompt_first" — one composed exam-setter prompt, one-shot LLM call, no
    topic-diversity/archetype/difficulty-audit machinery. Skips the exam-
    reference-brief LLM call entirely (wasted for this mode — nothing in
    the prompt uses it). FILE: examPromptFirst.service.js,
    examPromptComposer.service.js.

  "paper_reference" — a stored past-paper text file
    (firstedu-backend/reference-papers/<profile>.txt) is analyzed for a
    DIFFICULTY CALIBRATION ONLY (never a topic/slot source — topics stay
    100% AI-planned, identical to "default"). Injected into the solve-first
    skeleton prompt as a floor-to-exceed, not a target to replicate:
    "every question you write must be strictly MORE difficult than that
    observed pattern... do not use it to constrain which topics/slots you
    write about." FILE: referencePaperLibrary.service.js
    (loadReferencePaper, extractReferencePaperGuidance — returns only
    { difficultyCalibration }, nothing else).

PROVIDER FALLBACK
------------------
  aiQuestion.service.js: withProviderFallback() / dispatchGenerationLLM() /
  dispatchGenerationLLMText() wrap callQuestionBankGenerationLLM /
  callQuestionBankGenerationLLMText. On a provider-availability error
  (isProviderAvailabilityError — rate limits, 503s, quota exhaustion), the
  call is retried on the next provider in buildProviderFallbackChain()
  instead of failing the batch. Traced as PROVIDER_FALLBACK { from, to,
  reason }. Verified live 2026-07-13: a Gemini 429 (quota exhausted)
  fell through to OpenAI mid-batch without the caller noticing.

MULTIPLE-CHOICE "MULTIPLE" TYPE — EXACTLY 2 CORRECT ANSWERS
---------------------------------------------------------------
  Enforced in THREE places, all in aiQuestion.service.js unless noted:
    1. Generation prompt text: "correctAnswer is an array of EXACTLY 2
       letters" (not "up to 4").
    2. parseQuestionBankAIItem(): multipleCorrectIndexes.length must be
       exactly 2 — outside that range throws DroppableQuestionError (not a
       plain Error). The batch parser (parseQuestionBankAIItems) catches
       this per-item and DROPS just that one question instead of failing
       the whole batch; an undercount of "multiple" questions from dropped
       items is tolerated (an overcount still throws — that would mean a
       different bug). Same tolerance applied to the correctness-repair
       loop (applyRepairedQuestions).
    3. correctnessPreAudit.service.js: detectAllOptionsMarkedCorrect()
       flags questions where uniqueCorrect.size > 2 as a deterministic
       correctness issue (belt-and-suspenders against anything that slips
       past #1/#2, e.g. items built via a different path).

STEP F4 — SKELETON → MCQ (CODE, NO LLM)
---------------------------------------
FILE: firstedu-backend/src/services/questionSolveFirst.service.js

  skeletonsToQuestions(skeletons, tierSlots, conceptSlots, batchOpts, repairDeps)
    repairDeps carries { examProfile, subject } through to the mandate check
    below — this is what lets validateHardSkeletonMandate() apply the right
    rigor tier per exam instead of always assuming JEE.
    FOR each skeleton:
      validateHardSkeletonMandate(sk, tier, { examCalibrated, examProfile, subject })
                                             [hardQuestionMandate.service.js]
      verifySkeletonAnswer()                [questionSolveFirst.service.js]

          — finalAnswer.type "text" (non-STEM) uses buildTextDistractors(),
            the model's own distractorValues used directly as wrong options
            (no arithmetic offset synthesis, unlike numeric). Throws (routes
            to repair queue) if fewer than 3 distinct usable distractors.
          — hybridization detection is now ANCHORED to actual notation
            shape (/^sp[superscripts]?(?:d[superscripts]?)?$/i) with a
            type !== "text" guard. Was a loose /sp/.test(display) substring
            match that also matched ordinary English words containing "sp"
            ("response", "special", "disproportionation"), silently
            corrupting non-STEM text-answer options into chemistry
            notation whenever one happened to contain "sp".
        lockExplanationToMarkedOption()     — numbered Step 1: Step 2: ...
        assertBuiltMcqConsistency()         — validates answer-explanation consistency
          → NEW (2026-07-15): assertSolveStepsConsistency() performs deep
            validation of ALL intermediate computed values in _solveSteps
            against the marked answer, catching the "computes X, marks Y"
            failure mode where LLM solves correctly but marks wrong option.
            Extracts equations (v₂ = 8, P₂ = 170000), result statements
            (therefore k = 3), and verifies the final computed value matches
            the marked option within tolerance. Also detects contradictory
            intermediate calculations (k² = 11 followed by k = 3 instead of
            k ≈ 3.317). Questions failing this check are thrown to the repair
            queue instead of being built with wrong answer keys.
        assertGenerationCorrectness()
        Sets: questionText, options, correctIndex, explanation,
              _solveSteps, _conceptSlot

  AI_QB_MAX_SKELETON_REPAIR_CALLS) caps per-skeleton repair LLM calls per
  batch — items beyond the cap are deferred (traced SKELETON_REPAIR_CAPPED)

  IF batch < target after SOLVE_FIRST_MAX_ATTEMPTS:
