PART A — HOW TO REACH difficultyMatchScore 80+ (PRACTICAL GUIDE)
================================================================================

Your target metric is OpenAI validation difficultyMatchScore (not internal
finalize difficultyMatchScore). Internal finalize often scores 82–94 while
OpenAI validation stays ~64–66 when:

  1. Explanations contain meta draft text ("adjusting", "is incorrect", "Let's use")
  2. OpenAI auditor sees formula plug-ins despite multi-step _solveSteps
  3. A few questions per bank are flagged too-easy (Q1, Q4, Q6 pattern)
  4. Client validate payload may omit _solveSteps (only explanation text sent)

WHAT THE PIPELINE ALREADY DOES (AI STEERING — 3 LLM CALLS PER CHUNK)
--------------------------------------------------------------------
  Call 1 — Archetype planning (Gemini/OpenAI):
    resolveConceptArchetypeSteering()
      → buildArchetypePlanningPrompt()
      → callQuestionBankGenerationLLM()
      → parseArchetypePlanResponse()
    Plans N unique hard slots + per-slot blueprints for the batch.

  Call 2 — Skeleton generation (Gemini/OpenAI):
    buildSolveFirstSkeletonPrompt()
      → callQuestionBankGenerationLLM()
      → parseSolveFirstSkeletons()

  Call 3 — Validation audits (OpenAI default):
    buildTopicRelevancePrompt() + buildCorrectnessAuditPrompt()
      + buildJeeAuthenticityAuditPrompt()
    mergeValidationResults() → difficultyMatchScore

TO ASK AN AI TO FIX LOW DIFFICULTY — USE evaluation_regen (BUILT-IN)
--------------------------------------------------------------------
After validate returns difficultyMatchScore < 80:

  1. Frontend calls validate → gets regenerationInstructions + dimensionScores
  2. User triggers regenerate with:
       generateIntent: "evaluation_regen"
       topicRelevanceEvaluated: true
       topicRelevanceRegenerated: false
       topicRelevanceFeedback: { full validate response payload }
  3. Backend injects into generation prompt:
       buildRegenerationEscalationBlock()
       buildRegenerationQualityGatesBlock()
       formatTopicRelevanceFeedbackBlock()
  4. evaluation_regen uses **solve-first** for STEM singles (same as initial).
     No one-shot fallback on regen — partial batches merge client-side.
  5. Post-skeleton **LLM difficulty self-audit** rejects weak skeletons before MCQ build.
  6. **Targeted regen** replaces only failed question numbers (not full bank upscale).

ALTERNATIVE: ADD A 4TH AI CALL (IMPLEMENTED)
---------------------------------------------
  Post-skeleton, pre-finalize:
    applySkeletonDifficultySelfAuditGate() → LLM scores each skeleton 0–100
    applyDifficultySelfAuditGate() on finalize for built MCQs
  Env: AI_QB_DIFFICULTY_SELF_AUDIT=1 (default on), AI_QB_DIFFICULTY_SELF_AUDIT_MIN=75

ENV VARS THAT AFFECT DIFFICULTY
-------------------------------
  AI_QB_SOLVE_FIRST=1          — solve-first path (default on)
  AI_QB_ARCHETYPE_STEERING=1   — AI plans slots (default on; 0 = static catalog)
  AI_QB_ARCHETYPE_HISTORY=1    — persist archetypes per bank section (default on)
  AI_QB_DIFFICULTY_SELF_AUDIT=1 — skeleton + finalize LLM difficulty gate (default on)
  AI_QB_INITIAL_DIFFICULTY_TARGET=92
  OPENAI_CORRECTNESS_AUDIT_MODEL=gpt-4o
  GEMINI_TEXT_MODEL / OPENAI_QB_GENERATION_MODEL

STEP F4 — SKELETON → MCQ (CODE, NO LLM)
---------------------------------------

------------------------

SCORING — mergeValidationResults()
------------------------------------
FILE: firstedu-backend/src/services/topicRelevanceValidation.service.js

  computeDimensionalScores(confirmedIssues, {
    modelCorrectnessScore,
    authenticityAuditScore,      ← authenticity LLM difficultyScore
    deterministicDifficultyScore ← runDeterministicDifficultyAudit score
  })

  difficultyMatchScore formula when authenticity LLM provides difficultyScore:
    core = llm_difficulty * 0.4 + deterministic * 0.6  (if det available)
    difficultyMatchScore = core * 0.8 + penaltyFromIssues * 0.2

  blendValidationScores() → topic + correctness + authenticity/style
  applyValidationScoreCaps() — caps overall if difficultyMatch < 60 or < 40

  OUTPUT FIELDS:
    overallScore, topicRelevanceScore, correctnessScore, styleScore,
    authenticityScore, diversityScore, difficultyMatchScore,
    dimensionScores: { correctness, style, authenticity, diversity,
                       difficultyMatch, explanationQuality },
    confirmedIssues[], regenerationInstructions, verdict, pipelineSummary

THRESHOLDS
----------
  TOPIC_RELEVANCE_PASS_SCORE — pass bar for regen eligibility
  REGEN_TARGET_SCORE = 80
  INITIAL_GEN_DIFFICULTY_MATCH_TARGET = 80
  DIFFICULTY_MATCH_CAP_THRESHOLDS.LOW = 40, .MID = 60

FRONTEND TRIGGER
----------------
FILE: -firstedu-frontend/src/features/admin/question-bank/components/AiSuggestionModal.jsx

  handleValidateTopicRelevance() → stores eval + topicRelevanceWorkflowByProvider.evaluated = true

  EvaluationProviderPanel → "Regenerate failed questions" when canRegenerateFromTopicEvaluation()

    → generateIntent: 'evaluation_regen'
    → topicRelevanceFeedback: buildTopicRelevanceFeedbackPayload(relevance)
    → excludeQuestionTexts: passing stems only (collectPassingQuestionStems)
    → mergeTargetedRegeneration() merges replacements into existing bank by flawedQuestionNumbers

FILE: -firstedu-frontend/src/features/admin/question-bank/components/aiSuggestion.utils.js

  canRegenerateFromTopicEvaluation(), validateRegenerateFromEvaluationRequest()
  mergeTargetedRegeneration(), deriveTargetedRegenerationCounts()
  buildValidationSlotIndex(), collectPassingQuestionStems()

BACKEND REGEN FLOW
------------------
FILE: firstedu-backend/src/services/aiQuestion.service.js

  generateQuestionBankSuggestions({ generateIntent: "evaluation_regen", ... })
    assertGenerationWorkflowAllowed() — requires:
      topicRelevanceEvaluated === true
      topicRelevanceRegenerated === false
      topicRelevanceFeedback.overallScore present
      score < TOPIC_RELEVANCE_PASS_SCORE OR confirmed issues

    resolveTargetedRegenerationCounts()     [regenerationTargeting.service.js]
      → replacement count = failed question numbers only (not full bank upscale)

    shouldUseSolveFirstGeneration() → TRUE for STEM singles (same as initial)
    → applySkeletonDifficultySelfAuditGate() before MCQ build
    → NO one-shot fallback on regen deficit (partial batch returned for client merge)

    loadPersistedArchetypes() / persistArchetypes()  [archetypeHistory.service.js]

  Focused regen prompts (buildQuestionBankPrompt / buildSolveFirstSkeletonPrompt):
    buildRegenerationEscalationBlock()      [topicRelevanceValidation.service.js]
    buildRegenerationQualityGatesBlock()    [topicRelevanceValidation.service.js]
    Skips heavy calibration/JEE blocks on evaluation_regen

  Response includes targetedRegeneration: { replacementCount, failedQuestionNumbers }

AFTER REGEN
-----------
  Frontend merges replacements, sets topicRelevanceWorkflowByProvider.regenerated = true
  User re-evaluates merged bank — topicRelevanceRegenerated blocks second evaluation_regen
